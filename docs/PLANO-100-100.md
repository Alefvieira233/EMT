# Plano 100/100 — De ferramenta interna a produto comercial elite

> Documento gerado em 2026-04-18 como resposta à pergunta "qual a análise realista
> do plugin e como chegar a 100/100". É uma visão de arquiteto — opinativa,
> priorizada, com custos e definitions-of-done.
>
> **Score hoje:** 7/10 como ferramenta interna, 5/10 como produto público.
> **Meta:** 10/10 comercial público, pronto para Autodesk App Store e escala.

---

## Diagnóstico — onde o plugin está forte e onde sangra

**Forte hoje.** Padrão Command → Service → Model aplicado com rigor real (não só
documentação), `FerramentaCommandBase` elimina 80% do boilerplate típico de
Revit add-in, Logger com fallback para `Debug.WriteLine`, CHANGELOG mantido,
installer com 4 camadas de automação, 347 testes passando, AIOX pass recente
eliminou catch-blocks silenciosos e inconsistências culturais.

**Sangra hoje, em ordem de criticidade:**

1. **HMAC_SECRET hardcoded** em `Licensing/KeySigner.cs` — se o repo virar
   público ou a DLL for decompilada, qualquer um forja licença. Bloqueador
   absoluto para distribuição pública.
2. **Sem code signing** no .MSI — SmartScreen bloqueia, corporativos recusam
   instalar.
3. **Single version** (Revit 2025 apenas) — perde 50%+ do mercado que ainda
   usa 2024 e vai ficar irrelevante quando 2026 lançar.
4. **Sem abstração sobre `Document`/`UIDocument`** — serviços dependem direto
   da API Revit, testes de comportamento real são inviáveis, breaking changes
   do Revit viram caça-refactor em dezenas de arquivos.
5. **347 testes mas coverage real de Services é baixo** — quase tudo é teste
   de Config/POCO/validação, não de comportamento dos Services (porque não dá
   pra testar sem Revit, porque ver item 4).
6. **Nenhuma telemetria / crash report** — quando um cliente reclamar, você
   depende dele mandar o log manualmente.
7. **Sem auto-updater** — cada release é "cliente instala MSI na mão". Fricção
   alta, adoção de fixes lenta.
8. **Bus factor 1** — você é o único que entende o código. Se quiser vender
   seriamente, precisa de onboarding documentation + no mínimo um backup
   humano conhecendo o repo.

---

## Princípios do plano

1. **Cada fase tem Definition of Done testável.** Nada de "melhorar
   arquitetura" como item. É `coverage ≥ 70% em Services/`, ou
   `signtool verify /pa FerramentaEMT.msi retorna Verified`.
2. **Paralelismo onde possível.** Fase 2 (testes) e Fase 3 (CI/CD) rodam
   em paralelo após Fase 1 pronta. Fase 0 bloqueia tudo.
3. **Feature flags para mudanças invasivas.** DI container, IRevitContext,
   auto-updater — todos entram atrás de flag que desativa comportamento
   novo em caso de regressão.
4. **Zero regressão em comandos existentes.** Golden-master tests capturam
   output atual de ListaMateriais/Cotas. Qualquer refactor que muda o byte
   do Excel precisa ser intencional.
5. **Custo monitorado.** Meta de infra < US$ 1.000/ano.

## FASE 0 — Bloqueadores de release pública

**Duração:** 2 semanas. **Paralelizável:** não, tudo é sequencial aqui.

### 0.1 Externalizar HMAC_SECRET

**Imediato (1 dia):** Mover a constante para `Licensing/Secrets.generated.cs`
gerado em build-time a partir de env var `EMT_HMAC_SECRET`. Arquivo no
`.gitignore`. Em Debug, usa `"DEV_ONLY_DO_NOT_SHIP"` literal — se alguma
Release builder acidentalmente gerar sem a env, CI falha.

```xml
<!-- FerramentaEMT.csproj -->
<Target Name="GenerateSecrets" BeforeTargets="CoreCompile">
  <Error Condition="'$(Configuration)'=='Release' AND '$(EMT_HMAC_SECRET)'==''"
         Text="EMT_HMAC_SECRET env var obrigatória em Release" />
  <WriteLinesToFile File="Licensing/Secrets.generated.cs"
    Lines="namespace FerramentaEMT.Licensing%3B internal static class Secrets %7B public const string HmacKey = %22$(EMT_HMAC_SECRET)%22%3B %7D"
    Overwrite="true" />
</Target>
```

**Médio prazo (1 semana):** Licenças assinadas server-side via Cloudflare
Worker. Você mantém o secret no env da Worker; `EmtKeyGen` local é descontinuado.
Cliente compra, Worker gera chave assinada, email para cliente. Revocação vira
uma entrada na KV store da Worker (~US$ 5/mês).

**Longo prazo (opcional, fase 4):** Validação online periódica — a cada 7 dias
DLL faz `POST https://license.ferramentaemt.com/validate` com chave. Server
retorna `{valid: true, expires: "2027-04-01"}`. Permite kill switch.

**Definition of Done:**
```bash
strings FerramentaEMT.dll | grep -i "hmac\|secret" # não retorna nada sensível
```

### 0.2 Code Signing

**Compra:** Sectigo OV (~US$ 250/ano) ou DigiCert EV (~US$ 400/ano). **Prefira
EV** — elimina SmartScreen imediatamente em vez de depender de reputação.

**Armazenamento:** `.pfx` convertido em base64, armazenado em GitHub Secret
`CODESIGN_PFX_BASE64` + `CODESIGN_PASSWORD`.

**CI step:**
```yaml
- name: Sign MSI
  run: |
    echo "${{ secrets.CODESIGN_PFX_BASE64 }}" | base64 -d > cert.pfx
    signtool sign /f cert.pfx /p "${{ secrets.CODESIGN_PASSWORD }}" `
      /tr http://timestamp.digicert.com /td sha256 /fd sha256 `
      artifacts/FerramentaEMT.msi
    signtool verify /pa artifacts/FerramentaEMT.msi
```

**Definition of Done:** Clicar 2x no MSI em Windows 11 limpo não dispara
"Unknown publisher". `signtool verify /pa` retorna `Successfully verified`.

### 0.3 Multi-versão Revit (2024 / 2025 / 2026)

**Estratégia:** multi-target com `#if REVIT2024 / 2025 / 2026`.

```xml
<PropertyGroup>
  <TargetFrameworks>net48;net8.0-windows</TargetFrameworks>
  <DefineConstants Condition="'$(TargetFramework)'=='net48'">REVIT2024</DefineConstants>
  <DefineConstants Condition="'$(TargetFramework)'=='net8.0-windows'">REVIT2025;REVIT2026</DefineConstants>
</PropertyGroup>
```

**APIs divergentes conhecidas:**
- `ElementId.IntegerValue` (≤2023) vs `ElementId.Value` (2024+)
- `UnitUtils.ConvertToInternalUnits` signature mudou em 2022
- `Parameter.AsValueString` culture-aware em 2025

**Installer:** detecta Revits instalados (`%ProgramFiles%\Autodesk\Revit 20*\`)
e coloca .addin em cada pasta de Addins.

**CI matrix:**
```yaml
strategy:
  matrix:
    revit: [2024, 2025, 2026]
```

**Definition of Done:** Um zip de release instala e todos os 32 comandos rodam
em Revit 2024, 2025, 2026 sem crash.

## FASE 1 — Hardening arquitetural

**Duração:** 4 semanas. **Paralelizável com:** Fase 2 após 1.1 pronto.

### 1.1 IRevitContext — abstração sobre Document

O grande salto. Hoje:

```csharp
public class MarcarPecasService {
    public void Executar(Document doc, MarcarPecasConfig cfg) { ... }
}
```

Meta:

```csharp
public interface IRevitContext {
    IEnumerable<IStructuralElement> GetStructuralElements(ScopeFilter filter);
    ITransaction BeginTransaction(string name);
    ILevel? GetLevel(string name);
    IParameter? GetParameter(ElementRef elem, string name);
    Result SetParameter(ElementRef elem, string name, object value);
    // ~30 métodos cobrindo o que hoje os Services usam
}

public class RevitContext : IRevitContext {
    private readonly Document _doc;
    // impl real
}

public class FakeRevitContext : IRevitContext {
    // impl em memória para testes
    public List<FakeElement> Elements { get; set; } = new();
}

public class MarcarPecasService : IMarcarPecasService {
    private readonly IRevitContext _ctx;
    public MarcarPecasService(IRevitContext ctx) => _ctx = ctx;
    public Result<MarcaResult> Executar(MarcarPecasConfig cfg) { ... }
}
```

**Migração incremental:** 1 service por semana. Ordem sugerida, do mais crítico
para o menos:
1. ModelCheckService (já só lê, migração trivial)
2. MarcarPecasService
3. ListaMateriaisService
4. CotarPecaFabricacaoService
5. ... resto

**Pegadinha:** algumas APIs Revit não cabem bem em abstração (Geometry,
Transactions aninhadas). Para essas, `IRevitContext.GetRawDocument()` como
escape hatch documentado. Usar com parcimônia.

**Definition of Done:** Pelo menos 5 Services rodando testes com
`FakeRevitContext`, zero uso de `Document` direto nesses 5.

### 1.2 DI container

Microsoft.Extensions.DependencyInjection. Bootstrap em `App.OnStartup`:

```csharp
public class App : IExternalApplication {
    public static IServiceProvider Services { get; private set; } = null!;

    public Result OnStartup(UIControlledApplication app) {
        var sc = new ServiceCollection();
        sc.AddSingleton<ILogger>(Logger.Instance);
        sc.AddSingleton<ILicenseGate, LicenseGate>();
        sc.AddScoped<IRevitContext, RevitContext>(); // recriado por comando
        sc.AddTransient<IMarcarPecasService, MarcarPecasService>();
        // ... registrar todos os services
        Services = sc.BuildServiceProvider();
        return Result.Succeeded;
    }
}

public class CmdMarcarPecas : FerramentaCommandBase {
    protected override Result ExecuteImpl(...) {
        var svc = App.Services.GetRequiredService<IMarcarPecasService>();
        // ...
    }
}
```

**Por que vale:** adicionar um novo service deixa de ser "caçar todos os lugares
que precisam dele". Trocar impl real por mock em testes vira 1 linha.

### 1.3 Result<T> pattern

Parar de usar exceções para falhas de negócio.

```csharp
public readonly record struct Result<T> {
    public T? Value { get; init; }
    public string? Error { get; init; }
    public ErrorKind Kind { get; init; }
    public bool IsSuccess => Error is null;

    public static Result<T> Ok(T value) => new() { Value = value };
    public static Result<T> Fail(string error, ErrorKind kind = ErrorKind.Business)
        => new() { Error = error, Kind = kind };
}

public enum ErrorKind { Business, Validation, External, Bug }
```

Services retornam `Result<T>`. Commands decidem:
- `ErrorKind.Validation` → `TaskDialog` amigável
- `ErrorKind.Business` → warning log + dialog
- `ErrorKind.Bug` → crash report + "Envie o log"

**Exceções verdadeiras** (OutOfMemory, Revit API crash) continuam sendo
exceções — só falhas previsíveis viram Result.

### 1.4 IProgress + CancellationToken

Operações longas (ListaMateriais em 10k elementos, ModelCheck full) congelam
o Revit. Solução:

```csharp
public interface IListaMateriaisService {
    Task<Result<ListaResult>> Executar(
        ListaConfig cfg,
        IProgress<ProgressReport>? progress = null,
        CancellationToken ct = default);
}

public record ProgressReport(int Current, int Total, string Message);
```

WPF window mostra `ProgressBar` bindado + botão Cancelar. Usuário cancela
e cai no `Transaction.Rollback()`.

**Definition of Done:** ListaMateriais em projeto de 5k elementos mostra
progresso a cada 100 elementos e pode ser cancelada sem deixar transaction
aberta.

## FASE 2 — Pirâmide de testes

**Duração:** 6 semanas. **Paralelizável com:** Fase 3.

### 2.1 Unit tests com FakeRevitContext

**Meta:**
- Coverage ≥ 70% em `Services/`
- Coverage ≥ 90% em `Models/` (POCOs devem ser trivialmente 100%)
- Todo bug corrigido vira regression test

**Estrutura:**
```
Tests/
├── FerramentaEMT.Tests.Unit/
│   ├── Services/MarcarPecasServiceTests.cs
│   ├── Services/ListaMateriaisServiceTests.cs
│   ├── ... (um por service)
│   └── Fixtures/FakeRevitContextBuilder.cs
└── FerramentaEMT.Tests.Integration/  (Fase 2.2)
```

Ferramenta: xUnit + FluentAssertions + Moq. Builder pattern para
`FakeRevitContext` facilita teste: `new FakeContextBuilder().WithBeam("V1", ...).Build()`.

### 2.2 Integration tests dentro do Revit

**Framework:** [xUnit.Revit](https://github.com/paddor/xUnit.Revit) ou custom
runner via `JournalManager`.

**Estratégia custom (mais simples):** cada teste é um command que lê um `.rvt`
de fixture, executa lógica, asserta via `TaskDialog` ou log. Journal Revit
executa sequência de comandos e parseia output.

**Fixtures:** `Tests.Integration/Fixtures/` contém:
- `empty.rvt` — projeto vazio
- `small-estrutura.rvt` — 50 elementos simples
- `large-estrutura.rvt` — 2k elementos (benchmark)
- `edge-cases.rvt` — vigas sem nível, pilares sem marca, etc.

**CI runner:** self-hosted Windows com Revit 2025 instalado. ~US$ 50/mês em VPS
com Windows + licença educacional do Revit (gratuita para empresas < US$ 3M/ano
via Autodesk Education, ou licença comercial paga).

Alternativa mais barata: rodar integration tests só em pre-release, manualmente
na sua máquina. CI trava no unit level.

### 2.3 Golden master / snapshot tests

ListaMateriais gera `.xlsx`. Refactor pode mudar ordem de linhas, formatação,
sem parecer quebrar.

**Solução:**
```csharp
[Fact]
public async Task ListaMateriais_SmallEstrutura_DeveBaterComGoldenMaster() {
    var ctx = LoadFixture("small-estrutura.rvt");
    var result = await _svc.Executar(DefaultConfig(), ctx);
    var xlsxBytes = await File.ReadAllBytesAsync(result.OutputPath);

    var golden = await File.ReadAllBytesAsync("Fixtures/golden/small-lista.xlsx");
    SemanticXlsxComparer.AssertEqual(golden, xlsxBytes);
    // compara células + valores, ignora metadata de data/autor
}
```

Check-in do golden no repo. Se lógica mudar intencionalmente, regenera golden
(via `dotnet test --update-goldens`) e o diff do golden entra no PR.

### 2.4 Benchmark suite

BenchmarkDotNet em `Benchmarks/` projeto separado. CI falha se p95 piorar > 20%.

```csharp
[Benchmark]
public async Task ListaMateriais_10kElementos() {
    await _svc.Executar(Config, _largeCtx);
}
```

Baseline salvo em `benchmarks/baseline.json`. Comparação via `benchmarkdotnet-compare`.

### 2.5 Mutation testing (opcional, bonus)

[Stryker.NET](https://stryker-mutator.io/docs/stryker-net/introduction/) testa
qualidade dos testes mutando código e vendo se testes pegam. Meta: mutation
score ≥ 60% no primeiro ano.

## FASE 3 — CI/CD grade profissional

**Duração:** 3 semanas. **Paralelizável com:** Fase 2.

### 3.1 GitHub Actions matrix

```yaml
name: build
on: [push, pull_request]
jobs:
  build:
    runs-on: windows-latest
    strategy:
      matrix:
        revit: [2024, 2025, 2026]
        configuration: [Debug, Release]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with: { dotnet-version: '8.0.x' }
      - run: dotnet build -c ${{ matrix.configuration }} /p:RevitVersion=${{ matrix.revit }}
      - run: dotnet test -c ${{ matrix.configuration }} /p:RevitVersion=${{ matrix.revit }}
      - uses: actions/upload-artifact@v4
        if: matrix.configuration == 'Release'
        with:
          name: FerramentaEMT-R${{ matrix.revit }}
          path: artifacts/**/*.msi
```

6 jobs por PR, ~15 min cada, paralelos. Custo: grátis (GitHub Actions free tier
para repos privados pequenos).

### 3.2 Signed release workflow

`.github/workflows/release.yml` dispara ao tag `v*`:

```yaml
on:
  push:
    tags: ['v*']
jobs:
  release:
    # build matrix como em 3.1
    # + step de signing
    # + step de upload no GitHub Releases
    # + step de gerar SBOM
    # + step de notificar via email (Resend.com, 3k emails grátis/mês)
```

### 3.3 Conventional Commits + semantic-release

Todos os commits em `feat:`, `fix:`, `chore:`, `refactor:`, etc. Enforçado via
`commitlint` no husky/git hook + `.github/workflows/lint-commits.yml`.

`semantic-release` auto-detecta: `feat` bump minor, `fix` bump patch,
`BREAKING CHANGE:` bump major. Gera CHANGELOG entry, cria tag, dispara release.

### 3.4 SBOM + vulnerability scanning

- `CycloneDX .NET generator` → `sbom.json` no artifact
- Dependabot alerta NuGet vulnerabilities
- `dotnet list package --vulnerable` falha CI se CVE HIGH

### 3.5 Branch protection

Main:
- Require PR (mesmo que você auto-abra pra você mesmo)
- Require CI green
- Require linear history
- No force-push

Hoje é você solo — configure auto-approve para seus próprios PRs mas mantém
CI gate.

## FASE 4 — Observabilidade & operação

**Duração:** 4 semanas.

### 4.1 Crash reporter

**Opção A (grátis):** Sentry.NET free tier — 5k events/mês.

**Opção B (você controla):** POST JSON para Cloudflare Worker
→ grava em R2 bucket + envia email via Resend para seu inbox.

**Privacy:** nunca enviar nome de arquivo `.rvt`, nome de cliente,
caminhos absolutos. Só stack trace, versão do plugin, versão Revit,
OS, linguagem.

```csharp
AppDomain.CurrentDomain.UnhandledException += (s, e) => {
    var ex = e.ExceptionObject as Exception;
    CrashReporter.Report(ex, new {
        PluginVersion = typeof(App).Assembly.GetName().Version?.ToString(),
        RevitVersion = app.ControlledApplication.VersionNumber,
        OSVersion = Environment.OSVersion.ToString(),
        Command = _currentCommand ?? "unknown",
    });
};
```

### 4.2 Telemetria anônima (opt-in)

**Eventos úteis:**
- `command_started(name)` e `command_finished(name, duration_ms, result)`
- `ribbon_opened`
- `dialog_cancelled(dialog_name)`
- `trial_started / trial_expired`

**Opt-in real:** na primeira execução, dialog explicando e perguntando.
Não enviar nada se recusado.

**Tool:** PostHog self-hosted (grátis em VPS próprio) ou Plausible. Dashboard
interno mostra top-N commands por semana, tempo médio, taxa de falha.

**Valor:** descobre que ninguém usa `Desabilitar União (Vista)` e considera
remover, ou que `Cotar Peça` falha 8% do tempo por tipo X de perfil.

### 4.3 Auto-updater

**Velopack** (sucessor do Squirrel.Windows, 2024+, .NET 8 nativo).

Flow:
1. `App.OnStartup` dispara `UpdateManager.CheckForUpdatesAsync()` em background
2. Se versão maior existe em GitHub Releases, baixa no temp
3. Próxima abertura do Revit aplica update antes de carregar .addin

**Canal estável vs beta:** `--channel=stable` vs `--channel=beta` no manifest,
usuário opta nas config.

### 4.4 Diagnostic bundle

Botão no ribbon "Gerar Relatório de Diagnóstico":

```
FerramentaEMT-diagnostic-2026-04-18.zip
├── logs/
│   ├── emt-latest.log
│   ├── emt-previous.log
├── env.txt       (OS, Revit version, .NET version, culture)
├── addin-config.json
├── installed-families.txt  (só nomes, sem paths absolutos)
└── manifest.json
```

Usuário arrasta no email de suporte → você tem 90% do contexto sem fazer
back-and-forth.

## Métricas do "100/100" — checklist final

**Técnico:**
- [ ] CI matrix 2024/25/26 verde em todo PR
- [ ] Coverage ≥ 70% em Services/
- [ ] Integration tests rodando em self-hosted runner (ou local manual)
- [ ] Mutation score ≥ 60%
- [ ] Benchmarks com baseline; regressão > 20% falha CI
- [ ] Zero `catch { }` vazio; zero `double.Parse` culture-unsafe; zero path absoluto em .addin committed

**Segurança:**
- [ ] HMAC_SECRET fora do binário (`strings` não retorna)
- [ ] MSI com "Verified publisher" em Windows 11 limpo
- [ ] Licenças server-side com kill switch funcional
- [ ] SBOM gerado em toda release

**Operacional:**
- [ ] Auto-updater aplicou v1.2.1 → v1.2.2 em produção sem intervenção manual
- [ ] Crash reporter capturou ao menos 1 bug real e você corrigiu
- [ ] Dashboard de telemetria mostra top-5 commands semanais
- [ ] Diagnostic bundle funcional — 3 tickets de suporte resolvidos com ele

**Produto:**
- [ ] Docs site público recebendo > 100 visitas/mês
- [ ] i18n pt-BR + en-US funcionando
- [ ] 1 cliente pagante diferente de você em produção
- [ ] Listagem na Autodesk App Store aprovada
- [ ] NPS > 40 dos primeiros 10 clientes

**Governança:**
- [ ] Conventional Commits enforçado
- [ ] CHANGELOG automático
- [ ] Branch protection ligada
- [ ] Pelo menos 1 outra pessoa (dev contratado, sócio, ou mesmo parceiro técnico)
      capaz de dar um fix crítico em 24h se você ficar indisponível
